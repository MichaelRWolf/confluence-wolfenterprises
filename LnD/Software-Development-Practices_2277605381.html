A maturity model for a validation of a project – Jay Bazuzi

A maturity model for a validation of a project, rather than a binary "Should":0: No one knows how to comprehensively test the system

  • “We consistently ship defects we don’t know about.”

1: Expert(s) know how to comprehensively test the system

  • “Inexperienced engineers often write defects; experienced engineers/testers feel overburdened maintaining quality.”

2: The validation playbook is documented, so anyone can comprehensively test the system

  • “Testing takes such a long time, and each feature or bugfix requires tedious paperwork.”

3: Test execution is fully automated, but a human must interpret the result

  • “Test maintenance is expensive, requiring an army of specialists”

4: Test execution and interpretation is automated and reliable

  • “We release with confidence, but we don’t get to pick the cadence.”

5: Testing is automated, reliable, and fast

  • “We can release on any cadence. Anyone can run experiments in production. Unexpected discoveries are welcome.”

Refactoring

Arlo’s definitions - Refactoring, Remodeling, Editing, Micro-rewrite

Slack. (2022). Slack. Retrieved December 13, 2022, from Slack.com website: https://app.slack.com/client/TR3CD5C0J/CRG5YQFCG

I use:

  • Refactoring: a single, named transformation of the program with evidence that it is known not to alter any behavior of the program, whether that behavior is known or not.

  • Realistically, the only good ways to get this depend on static analysis by rules of the language, and generally happen at the AST level within a compiler. That can be done in an IDE or by using a recipe that lifts any error to be a type error.

  • Each refactoring makes a single, atomic design change, with no other effect on the system.

  • Remodeling: a transformation of the program with the intent that it does not alter any relevant behavior of the system.

  • Relevant is up to developer interpretation.

  • Intent is not supported by conclusive evidence, though it may be supported by partial evidence (such as a comprehensive set of unit tests).

  • Editing: changing code by editing its text, sometimes intending to change behavior and sometimes not.

  • Micro-rewrite: replacing a small chunk of code with another. This is often a remodel performed via editing.

Other’s definitions

Current refactoring defs are:

  • mine: as above.

  • Fowler's: a change in code that:

  • Is named, and performs nothing other than what that named refactoring declares.

  • Is performed via the recipe for that refactoring, or something with a greater level of safety.

  • Generally meets the requirements of what I call a Remodeling.

  • Depends on comprehensive tests for safety, and is performed with those comprehensive tests.

  • general public: very close to my definition of remodeling, but may include an arbitrary set of changes all at once. Also refers to the activity of changing a produc tin arbitrary ways with the intention of not messing anything relevant up.

As some examples,

  • Rename in Visual Studio for .Net is: a Refactoring under Arlo and Fowler's definitions, also a Remodeling, but not a micro-edit.

  • Internally it resolves compiles the code text far enough to get the symbol table, then looks up all inbound references in that table. It then replaces them all semantically on the AST, then back-generates the code. Then it re-compiles all impacted code and compares the resulting symbol table to the original, to make sure the text change didn't alter any name resolutions (eg, scope interactions).

  • Rename in JS in VSCode is: not a Refactoring under Arlo's definition, but can be under Fowler's if the code is sufficiently tested, also a Remodeling, but not a micro-edit.

  • Internally, it it performs a text search for duplicate text (including inside strings and comments), then attempts to resolve them via language heuristics. It then replaces those it feels sufficiently likely to reference the same name.

  • Rename by using search and replace is: not a refactoring under Arlo or Fowler, may be a Remodeling if it is not conflated with other changes, and may be a micro-edit depending on how it is performed.

  • Rename by hand-typing the new name multiple times is: not a refactoring, not a remodeling, definitely a micro-edit.

Automate Yourself with Checklists

We've been working on some ideas that connect with "Automate Yourself with Checklists".First, every project should have a playbook for:

  • Validating an arbitrary change

  • Releasing to production

  • Ongoing maintenance

And these playbooks should be written in a way that a non-expert can carry them out.Second, each playbook should say how long it takes to execute, for a non-expert. This gives management a visibility (into costs that are already silently being paid) and the opportunity to manage ("Wow, we are spending a lot on X; let's invest in that.")
