Wolf Enterprise : Proposal to eliminate incorrect formatting of phone numbers in Daylite

Created by Michael R. Wolf, last modified on 2020-01-14

Summary

See: <a href=”file:////Users/michael/Marketcircle/intl_phone.org”>Org file with Marketcircle proposal</a>

Daylite’s phone number reformatting is helpful for NANP phone numbers. For many 10- and 11-digit numbers, not only is the formatting against local conventions, but the reformatting translates the dialing sequence to be incorrect.

Examples:

  • NANP 10- (or 11-) digit number

    • “5095551212” (or “509 555 1212” or “509-555-1212”) → “(509) 555-1212)”

    • “15095551212” (or “1 509 555 1212” or “1 509-555-1212”) → “(509) 555-1212)”

      • Formatting: User formatting → conventional formatting (helpful or not?)

      • FaceTime.app or iPhone: Working → Working

  • (NANP) Helpful + Correct:

    • “+15095551212” (or “+1 509 555 1212” or “+1 509-555-1212”) → “(509) 555-1212)”

      • Formatting: User input → conventional

      • FaceTime.app or iPhone: Working (NANP + Internationally)
        → Working (NANP) + Not working (Internationally)

  • (Denmark) Unhelpful + Incorrect:

    • “+43 3084 8344” → “(453) 084-8344”.

      • Formatting: Conventional → Not conventional

      • FaceTime.app or iPhone: Working
        → Not working (in Denmark) + Not working (internationally)

I propose a simple change that would allow DL to:

  • follow conventions for NANP phone numbers

  • not be wrong for international numbers (though it may not be correct or follow conventions)

The logic is simple:

  • If the user enters a character string beginning with “+”, leave the string unchanged. Assume that the user knows the conventions for formatting a phone number. Therefore, the code need not attempt to enforce the conventions.

  • Otherwise, leave the code unchanged so that it reformats (or preserves) user input as it currently does.

  • (Optional) If the character string begins with “+1”, indicating a NANP phone number, allow the code to enforce the conventional formatting of “+1 (905) 480-5555”.

I propose that all telephone number reformatting code be disabled for character strings beginning with “+”, but remain enabled for 7-, 10-, and (maybe) 11-digit phone numbers.

Background

At the root of the issue is that the code assumes the input string of characters should be interpreted as a NANP phone number.

By convention, international telephone numbers are indicated in listings by prefixing the country code with a plus sign (+). https://en.wikipedia.org/wiki/Telephone_numbering_plan
